MySQL开发实战笔记

发表于 2019-03-28 更新于 2026-03-21
MySQL开发中常见的问题与解决方案，包含索引优化、事务处理、死锁分析等

本文记录了 MySQL 开发中常见的问题与解决方案，包括索引优化、事务处理、死锁分析等实战经验。

DDL 与 DML

类型 说明
DDL 数据定义语言，如 CREATE、ALTER、DROP 等，会超时
DML 数据操作语言，如 SELECT、INSERT、UPDATE、DELETE，会阻塞直到执行完成

优化技巧

查询语句优化

  1. 尽量用 JOIN 查询，左表是小表去 LEFT JOIN
  2. 效率低的场景：能用 EXISTS（小表）就不用 IN（大表）
  3. 用 SQL_NO_CACHE 查看执行计划，排除缓存干扰

索引优化

  1. 组合索引 (a,b,c) 查询只要带上 a 就走索引
  2. 组合索引 (a,b,c) 和 (a,c,d) 可以合并为 (a,b,c,d)
  3. 唯一组合索引跟普通组合索引用法一样，只是多了唯一约束
  4. 如果一个字段包含很多重复值，走索引意义不大，几乎是全表扫描

大数据量操作

当需要 UPDATE/DELETE 大量数据时，为避免大事务问题：

  1. 方案一：把匹配条件的记录 ID 查询出来，放入 Redis，用游标分批处理
  2. 方案二：查询出最大 ID 和最小 ID，分段处理

当需要SELECT时需要防止没有带条件查询所有记录，导致内存溢出。

事务与并发问题

事务未提交导致查询不到数据

问题场景：流程落地后需要推送数据到核算系统，但推送线程查询不到刚落地数据。

原因：启动线程太快，事务还未完全提交就开始查询。

死锁问题

死锁产生原因

锁只有在竞争一个以上的资源时才会产生，MySQL 死锁都是基于索引的。

典型死锁场景

因为表里建了唯一索引，INSERT 之前会锁住索引页，两个事务没提交时会互相死锁。

死锁日志分析

如何找到死锁日志

MySQL 发生死锁时，会自动检测并回滚其中一个事务，同时在错误日志中记录死锁信息。

查看方式：
方法一：查看最近一次死锁信息
SHOW ENGINE INNODB STATUS;

方法二：查看历史死锁日志
在 MySQL 配置文件中添加：
innodb_print_all_deadlocks = 1
在MySQL中看错误日志的位置：
SHOW VARIABLES LIKE ‘log_error’;
找到错误日志并查看 MySQL 错误日志文件:
cat error.log

死锁日志结构

死锁日志通常包含以下部分：

  1. TRANSACTION 信息：每个事务的信息，包括事务ID、活跃时间、SQL语句
  2. WAITING FOR THIS LOCK TO BE GRANTED：事务正在等待的锁
  3. HOLDS THE LOCK(S)：事务持有的锁
  4. WE ROLL BACK TRANSACTION：回滚的事务

常见锁类型及含义

锁类型 含义
lock_mode X 排他锁（Exclusive Lock），阻止其他事务读写
lock_mode S 共享锁（Shared Lock），允许其他事务读但不能写
lock_mode X locks rec but not gap 记录锁，锁住具体行记录
lock_mode X locks gap before rec 间隙锁（Gap Lock），锁住记录之间的间隙
lock_mode X insert intention 插入意向锁，INSERT 操作前获取
RECORD LOCKS 行级锁，锁住具体的行记录
lock_struct(s) 锁结构的数量
row lock(s) 行锁的数量
undo log entries undo 日志条目数

死锁日志示例

日志解读：

  • 事务1 正在执行 INSERT，正在等待获取插入意向锁
  • 事务2 持有共享锁（S锁），同时在等待另一个位置的共享锁
  • 事务1 和 事务2 形成循环等待，MySQL 检测到死锁，回滚了事务1

MySQL 基础架构

Server 层 vs 存储引擎层

MySQL 整体分为两层：

  • Server 层：连接器、查询缓存、分析器、优化器、执行器
  • 存储引擎层：负责数据的存储和提取（插件式，支持 InnoDB、MyISAM 等）

执行流程

  1. 连接器：管理连接权限，定期断开长连接
  2. 查询缓存：表更新会导致缓存失效，建议关闭
  3. 分析器：词法分析和语法分析
  4. 优化器：决定使用哪个索引、多表连接顺序
  5. 执行器：表权限检查，调用存储引擎接口

日志系统

redo log（重放日志）

作用：InnoDB 特有，保证 crash-safe 能力

特点：

  • 物理日志，记录”在某个数据页上做了什么修改”
  • 循环写，空间固定
  • 固定大小（如 4 个 1GB 文件）

原理：相当于”粉板”，write pos 记录当前位置，checkpoint 记录擦除位置。

InnoDB 的 redo log 是固定大小的，比如可以配置为一组 4 个文件，每个文件的大小是 1GB，那么这块“粉板”总共就可以记录 4GB 的操作。从头开始写，写到末尾就又回到开头循环写，如下面这个图所示:

write pos 是当前记录的位置，一边写一边后移，写到第 3 号文件末尾后就回到 0 号文件开头。checkpoint 是当前要擦除的位置，也是往后推移并且循环的，擦除记录前要把记录更新到数据文件。

write pos 和 checkpoint 之间的是“粉板”上还空着的部分，可以用来记录新的操作。如果 write pos 追上 checkpoint，表示“粉板”满了，这时候不能再执行新的更新，得停下来先擦掉一些记录，把 checkpoint 推进一下。

binlog（归档日志）

作用：Server 层日志，用于数据恢复和主从复制

特点：

  • 逻辑日志，记录原始逻辑（如”给 ID=2 加 1”）
  • 追加写，不会覆盖
  • 可用于数据恢复

总结

  1. 查询优化：多用 JOIN，合理使用索引，避免全表扫描
  2. 大事务：分批处理，避免长时间锁表
  3. 并发问题：注意事务提交时机，避免查询不到刚写入的数据
  4. 死锁分析：基于索引产生，分析锁等待关系
  5. 日志理解：redo log 保证持久化，binlog 用于恢复和复制
